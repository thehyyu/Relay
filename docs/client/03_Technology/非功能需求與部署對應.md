# 非功能需求與部署對應

> 同一份業務需求，NFR 指標不同，乙方選型就不同。
> 這張表說明每個 NFR 如何驅動部署版本的選擇。

## NFR 對比矩陣

| NFR | 指標 | v1 monolith | v2a microservices | v2b K8s | v3 hybrid |
|-----|------|:-----------:|:-----------------:|:-------:|:---------:|
| 回答延遲 | 秒 | < 30s | < 20s | < 15s | < 10s |
| 並發使用者數 | 人 | 1 | 10 | 50 | 彈性（burst） |
| 單點故障容忍 | 是/否 | 否 | 部分 | 是 | 是 |
| 服務獨立重啟 | 是/否 | 否 | 是 | 是 | 是 |
| 知識庫資料持久性 | 重啟後保留 | 本地檔案 | Docker volume | PVC (StatefulSet) | 雲端儲存 |
| 自動擴縮容 | 是/否 | 否 | 否 | 手動 (replicas) | 是（serverless） |
| 部署環境 | — | 本機 | 本機 Docker | minikube | 雲端 + 本機混合 |
| LLM 來源 | — | Ollama（本機） | Ollama（本機） | Ollama（本機） | Claude API（雲端） |
| 資料離開公司網路 | 是/否 | 否 | 否 | 否 | 是（v3 才開始） |

## 每個 NFR 的業務理由

**回答延遲**
甲方 IT 主任王建國要求：顧問不應因等待 AI 回答而中斷思路。
- v1 容忍較長延遲（學習環境）
- v3 要求 < 10s（接近即時對話體驗）

**並發使用者**
甲方初期試點 1 人（v1），擴大到部門（v2a），全公司上線（v2b），彈性高峰（v3）。

**單點故障容忍**
v1 單 script 掛掉即服務中斷，可接受（測試環境）。
v2b 起要求任一 Pod 掛掉後 60 秒內自動恢復（正式上線條件）。

**資料離開公司網路**
IT 主任的硬性要求：v1 ~ v2b 所有 LLM 推論必須在本機完成（Ollama）。
v3 開始使用 Claude API，需甲方另行審核資安政策。

## 部署選型決策摘要

| 版本 | 觸發 NFR | 乙方選型理由 |
|------|----------|-------------|
| v1 | 單人使用、低延遲容忍 | 最簡架構，驗證 agent loop 正確性 |
| v2a | 多人使用、需獨立重啟 | 服務拆分，API 契約明確 |
| v2b | HA 要求、資料持久性 | K8s StatefulSet 保護 ChromaDB |
| v3 | 彈性並發、更低延遲 | Serverless 處理短任務，Claude API 提升品質 |
