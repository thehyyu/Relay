# 需求規格

## 系統類型

- [x] 自建系統　說明：Multi-agent research assistant，從 raw API 自行組裝

## 架構模式（依部署版本演進）

- [x] 單體式（Monolithic）→ v1
- [x] 微服務（Microservices）→ v2a、v2b
- [x] 混合式（Hybrid serverless）→ v3

## 功能需求清單

| 需求編號 | 功能描述 | 優先級 | 狀態 |
|----------|----------|--------|------|
| FR-001 | 使用者可以用自然語言輸入問題 | 高 | 確認 |
| FR-002 | 系統從知識庫找出最相關的文件段落（語意搜尋） | 高 | 確認 |
| FR-003 | 回答必須附上來源文件名稱與引用段落 | 高 | 確認 |
| FR-004 | 知識庫外的問題，系統明確回答「無相關資料」 | 高 | 確認 |
| FR-005 | 回答格式結構化，易於閱讀（非流水文字） | 中 | 確認 |
| FR-006 | 知識庫可由管理員手動更新語料 | 中 | #待確認 更新頻率與流程 |
| FR-007 | 系統紀錄每次查詢與回答（audit log） | 低 | #待確認 是否為 Phase 1 需求 |

## 非功能需求

詳見 [[03_Technology/非功能需求與部署對應]]（各部署版本的 NFR 對比表）。

摘要：
- **效能**：回答延遲依部署版本不同，v1 目標 < 30s，v3 目標 < 10s
- **可用性（SLA）**：v1 無 SLA，v2b 起要求單 Pod 故障後 60 秒內自動恢復
- **擴展性**：v1 單一使用者，v3 支援彈性並發
- **資料持久性**：ChromaDB 向量資料庫重啟後必須保留
