# 風險登錄簿

## 風險清單

- R-001｜**LLM 幻覺（Hallucination）**
  - 衝擊：高 / 可能性：高
  - 說明：LLM 可能產生看似合理但無來源支撐的回答，顧問誤信後影響諮詢品質
  - 對策：強制要求 Write Agent 只能使用 Search Agent 回傳的段落，禁止憑空生成；測試集驗收時專門測「知識庫外問題」
  - 負責人：Hubert

- R-002｜**資料離開公司網路（v3 風險）**
  - 衝擊：高 / 可能性：中（v3 才出現）
  - 說明：v3 改用 Claude API，查詢內容與知識庫段落會送至外部 API，違反甲方 IT 政策需另行審核
  - 對策：v3 上線前提交資安評估報告，取得 IT 主任（王建國）書面核准；考慮對 query_text 做脫敏處理
  - 負責人：Hubert + 王建國

- R-003｜**ChromaDB 資料遺失（v1/v2a 風險）**
  - 衝擊：中 / 可能性：中
  - 說明：v1 資料存本地目錄，v2a 存 Docker volume，容器或機器異常可能導致向量資料遺失，需重新 embedding
  - 對策：v2b 起改用 K8s PVC，定期備份語料原始檔（`.md`）；v1/v2a 阶段接受此風險（學習環境）
  - 負責人：Hubert

- R-004｜**Ollama 服務不穩定（Mac mini 單點）**
  - 衝擊：高 / 可能性：低
  - 說明：v1 ~ v2b 全部 LLM 推論依賴同一台 Mac mini，機器重啟或網路中斷即服務全斷
  - 對策：v2b 加入 Ollama liveness probe；v3 改用 Claude API 消除此風險
  - 負責人：Hubert

- R-005｜**語料品質不一影響搜尋精準度**
  - 衝擊：中 / 可能性：高
  - 說明：初始語料為 Markdown 技術文件，若甲方後續匯入格式混亂的 PDF，切片品質將影響 embedding 效果
  - 對策：制定資料品質規則（見 [[04_Data/主數據清單]]），PDF 匯入前需人工確認
  - 負責人：林雅婷（甲方）+ Hubert（工具）
