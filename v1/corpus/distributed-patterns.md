# 分散式系統設計模式

## Circuit Breaker（斷路器）

當一個服務持續失敗時，斷路器會「跳開」，後續請求直接回傳錯誤，不再嘗試連接失敗的服務。

三個狀態：
- **Closed（閉合）**：正常運作，請求通過。
- **Open（斷開）**：服務失敗達到閾值，直接拒絕請求。
- **Half-Open（半開）**：嘗試少量請求測試服務是否恢復。

## Saga Pattern

跨多個服務的長事務處理模式，將大事務拆分為一系列本地事務，每步失敗時執行補償事務回滾。

兩種實現方式：
- **Choreography**：各服務監聽事件自行觸發下一步，去中心化。
- **Orchestration**：由中央協調器控制執行順序，邏輯集中。

## Sidecar Pattern

將輔助功能（日誌收集、服務發現、TLS 終止）部署在獨立容器中，與主容器並排運行在同一 Pod 內，不影響主應用程式邏輯。

## API Gateway

統一的入口點，負責：
- 路由（將請求導向對應的微服務）
- 認證授權
- 速率限制
- 請求/回應轉換
- 負載均衡

## Strangler Fig Pattern

將單體應用逐步遷移到微服務的策略：在舊系統前方加入代理層，逐步將功能轉移到新服務，直到舊系統完全被取代。

## 負載均衡策略

- **Round Robin**：輪流分配請求到各節點。
- **Least Connections**：分配到當前連線數最少的節點。
- **Consistent Hashing**：相同的請求參數固定路由到同一節點，適合快取場景。
