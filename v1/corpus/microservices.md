# 微服務架構

微服務架構將應用程式拆分為多個小型、獨立部署的服務，每個服務負責單一業務功能。

## 與單體架構的比較

| 面向 | 單體架構 | 微服務架構 |
|------|---------|-----------|
| 部署 | 整體部署 | 各服務獨立部署 |
| 擴展 | 整體水平擴展 | 個別服務獨立擴展 |
| 技術選型 | 統一技術棧 | 各服務可選不同技術 |
| 複雜度 | 程式碼複雜 | 運維複雜 |
| 故障隔離 | 一點故障影響全體 | 故障隔離在單一服務 |

## 核心原則

- **單一職責**：每個服務只做一件事，邊界清晰。
- **去中心化資料管理**：每個服務擁有自己的資料庫，不共用。
- **透過 API 溝通**：服務之間透過 REST 或 gRPC 呼叫，或透過訊息佇列非同步通訊。
- **故障設計**：假設服務會失敗，使用 Circuit Breaker、Retry、Timeout 等模式處理。

## 常見挑戰

- **分散式追蹤**：一個請求跨越多個服務，需要 distributed tracing（如 Jaeger、Zipkin）。
- **服務發現**：服務動態上下線，需要 service registry（如 Consul、Kubernetes Service）。
- **資料一致性**：跨服務的事務處理複雜，通常採用 Saga Pattern。
- **網路延遲**：服務間呼叫比本地呼叫慢，需謹慎設計介面粒度。
